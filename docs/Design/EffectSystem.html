EffectSystem

効果システム検討

概要

本ドキュメントは、YAML によって定義される「オブジェクト間の影響関係」の設計を検討した結果をまとめたものです。 GameElementDefinition.md が定める tick 駆動・範囲による状態決定・「ハードコードしない」「汎用エンジンに任せる」「ファイル追加だけで拡張できる」という方針を前提に、 装備・アイテム・キャラクターパラメータ・コンディションなどが互いに影響し合う仕組みを、単一の記法・単一の実行原理でどう表現するかを整理します。

本ドキュメントはあくまで現時点の検討結果であり、確定仕様書ではありません。文中に登場する防具・細菌感染・耐久値などの具体例は、 記法を説明するためのサンプルであり、実際のゲームバランスや病気システムそのものを規定するものではありません。

1. オブジェクトの親子関係とスロット

  • オブジェクト間の所属関係はツリー構造で表現します。1 つの子オブジェクトは必ず 1 つの親に属し、複数の親を持つ関係は採用しません。
  • 親オブジェクトは複数の「スロット」（配列）を持ちます。例: inventory（所持品）、equip（装備）、fixtures（部屋の備え付け設備）。
  • 子オブジェクトは held_in のような逆引き用のキャッシュフィールドを持ちますが、正の情報源はあくまで親側のスロット配列です。
  • スロット間の移動は move_to_slot という単一の汎用操作に集約します。親側スロット配列と子側の逆引きキャッシュの整合性は、この操作の中でのみ保証します。個別の YAML 定義側が両者を直接書き換えることは想定しません。
  • スロットに「種類タグ（kind）」は設けません。equip はそれ自体が唯一の装備用スロットであり、スロット名を直接参照します。スロットの意味をタグで抽象化するよりも、名前をそのまま参照する方が単純で、現時点で複数の装備スロットを区別する必要も生じていないためです。

2. 装備の排他制御（layer / covers）

  • equip スロットには複数のアイテムを同時に格納できます。排他制御はスロットの数ではなく、アイテムが持つ属性によって行います。
  • アイテムは covers（覆う部位のタグの配列。例: [torso]、[legs, waist]）と layer（重ね着の階層。例: base、outer）を持ちます。
  • 競合判定は、同じ (covers要素, layer) の組が、既に equip スロットに入っている他アイテムと重複する場合に装備不可とします。
    • 例: シャツ（torso, base）とジャケット（torso, outer）は共存可能です（layer が異なるため）。
    • 例: シャツ（torso, base）とつなぎ（legs/waist/torso, base）は共存不可です（torso, base が競合するため）。
  • 競合解決は「ブロック型」を採用します。競合するアイテムがある場合、プレイヤーが先に既存の装備を外す必要があります（自動退避は採用しません）。装備の入れ替えを明示的な操作として要求することで、プレイヤーが装備状態を把握しないまま暗黙に変更されることを避けるためです。

3. 効果（effects）の基本構造

効果は次の 3 つのレベルで定義でき、いずれも同一の記法・実行原理を用います。

  1. オブジェクトレベル（例: 防具そのものが持つ複数の恩恵の束）
  2. プロパティレベル（例: アイテムの重量プロパティが持ち主の負荷に寄与する）
  3. プロパティのステージレベル（例: 耐久値の残量帯によって防御効果が変わる）

記法は「発生条件を外側に、効果本体を effects: の中に」という形に統一します。effects: の配列要素の中に when を書くのではなく、条件キーの下に effects をぶら下げる構造です。

effects:
  - when: equip
    target: parent
    modify:
      defense: 5
      speed: 3
      accuracy: 2

トリガーの種類は以下の 2 つです。

  • when: <スロット名> … そのスロットに入っている間、継続的に有効（レベルトリガー）
  • on: <アクション名> … 特定のアクション発生時に一度だけ発火（エッジトリガー）。詳細仕様は別途検討中であり、本ドキュメントでは語彙の存在のみを記載します。詳細は今後追記します。
  • 条件キーを省略した場合は「常時（無条件）」を意味します。

target は効果の対象を指します。現時点で定義するのは self（自分自身）・parent（親）・child（子）の 3 つです。 ancestor（先祖）・sibling（兄弟）・descendant（子孫）は、必要性が生じた時点で改めて検討します。

4. 効果の種別（modify / add / lifecycle）

3 種類の効果種別を区別します。この区別は「値の可逆性」と「適用頻度」という 2 軸に基づきます。

  • modify: 条件が真である間だけ計算式に寄与する、可逆的な修飾子です。条件が偽になった瞬間に寄与は自動的に消えます。値そのものを書き換えるのではなく、実効値を都度導出するための入力として扱います。
    • 同一プロパティに対する複数の modify は単純加算で合成します。
    • 合成結果の上限・下限（min/max）はプロパティ（スタット）側の定義に持たせます。
  • add: 対象プロパティの実体値を直接、不可逆に加減算します。
    • on（エッジトリガー）配下の add は、そのアクション発生時に一度だけ加算します。
    • when（レベルトリガー）配下の add は、条件が真である tick 毎に加算します（例: 出血中の血液量減少、耐久値の毎 tick 減少）。
    • 「on なら一回、when なら毎 tick」という対応は暗黙のルールとして確定した仕様です。
  • lifecycle: プロパティの値ではなく、オブジェクトそのものの存在を操作します（例: destroy でオブジェクトを消滅させる）。modify/add が値操作であるのに対し、lifecycle は存在操作を担う第三のカテゴリとして区別します。

5. パラメータの段階（stages）

  • 数値パラメータ（プロパティ）は stages を持つことができ、現在値がどの段階にあるかによって、有効になる effects が切り替わります。
  • 段階の区間定義は min（下限）のみを指定します。上限は明示せず、次の段階の min によって自動的に決まります（半開区間、[min, 次の min より手前]）。この方式により、区間の隙間や重複が構造的に発生しません。
  • 最下段の段階は min: null（または省略）とし、それより下の残り全ての値を拾います。
  • 現在値に基づいて常に一意に段階を決定します。ヒステリシス（上昇時・下降時で閾値をずらす仕組み）は採用しません。プレイヤーが常に現在の状態から結果を逆算できるようにし、状況分析をしやすくするための設計意図です。
  • ステージの effects はレベルトリガー（そのステージにいる間ずっと有効）を基本とします。ステージ切替の瞬間だけ発火する仕組み（edge-triggered、例えば on_enter 的なもの）は将来検討事項として保留中です。

6. サンプル

以下は、ここまでの記法を用いた説明用のサンプルです。実際のゲームバランスを規定するものではありません。

防具（オブジェクトレベルの効果、複数の恩恵の束）

item: armor_leather
covers: [torso]
layer: base
effects:
  - when: equip
    target: parent
    modify:
      defense: 5
      speed: 3
      accuracy: 2

耐久値（プロパティのステージレベル、毎 tick 減少と消滅条件）

properties:
  durability:
    value: 100
    effects:
      - target: self
        add:
          durability: -1
    stages:
      - name: intact
        min: 1
      - name: broken
        min: null
        effects:
          - target: self
            lifecycle: destroy

細菌感染（オブジェクト化された「条件」、段階に応じた多段の影響）

感染度自体をキャラクターの生パラメータにはせず、conditions のようなスロットに格納される独立したオブジェクトとして扱います。 怪我・疲労などの「条件」は複数同時に存在しうるため、条件ごとに独立したオブジェクトとして表現する方が、キャラクター側のパラメータを増やし続けるより見通しが良いためです。

感染オブジェクトの progress プロパティが stages を持ち、段階ごとに体温・水分・意識などの他プロパティへ影響します。

condition: infection
properties:
  progress:
    stages:
      - name: none
        min: 0
      - name: mild
        min: 20
        effects:
          - target: parent
            add:
              temperature: 0.1
      - name: feverish
        min: 50
        effects:
          - target: parent
            add:
              temperature: 0.2
              hydration: -1
