ActionSystem

カード間の相互作用（actions / combinations の使い分け）

概要

本ドキュメントは、プレイヤーがカードに対して行う操作を、actions（メニュー型）と combinations（ドラッグ型）の どちらで表現すべきかの判断指針をまとめたものです。両者の文法自体は GameElementDefinition.md 11 節・12 節を 参照してください。

本ドキュメントは検討結果であり、確定仕様書ではありません。未決事項は 4 節に整理しています。

1. 使い分けの基準

すべてのアクション・組み合わせには、actor（プレイヤーキャラクター）が暗黙的に参加します。そのため 「2つの参加者が要るように見える操作」でも、もう一方が常に actor であれば actions（メニュー型）で表現できます。 eat（食べ物 self ＋ actor）だけでなく attack（攻撃対象の敵 self ＋ actor）もこれに当たります。

combinations（ドラッグ型）が必要になるのは、actor とは別の、道具や素材など特定のカードそのものを指定する 必要がある場合（「斧で木を切る」）に限られます。

  • メニュー型（actions）: 1枚のカードだけで完結する操作。カードを選択すると実行可能なアクションがボタンで 表示され、クリックすると実行できます。ロケーションカード限定の「探索」ボタンも、この特殊なケースです。
  • ドラッグ型（combinations）: カードを、actor ではない別の特定のカードにドラッグ＆ドロップして行う操作。

2. combinations のキー設計

combinations は object_defs/traits/props/actions と同様、識別子をキーとする辞書型ですが、 キーはマッチング対象ではなく、このドラッグ操作自体に付ける名前です。マッチング対象は with で別途指定します （GameElementDefinition.md 12.1 節）。キーを名前として自由にした結果、同じ with を指す複数のキーを書けてしまう 余地が残ります（例: wood に対して with: axe_tool の chop と hack を両方定義する）。この場合にどちらが実際に 選ばれるかの解決規則は未決定です（4 節）。

3. 対称的な組み合わせ

「棒とロープを組み合わせて槍を作る」のように、どちらのカードをどちらにドラッグしても成立してほしい、方向が 自明でない組み合わせは、両方のカードに combinations を書きます（GameElementDefinition.md 12.3 節）。 片方にしか書かなければ、その向きにドラッグしたときしか成立しません。

4. 未決事項・今後の検討課題

  • 実行エンジン（Assets/Scripts/Runtime/InteractionExecutor.cs）は、with に一致する combinations を 宣言順にすべて列挙する FindMatchingCombinations を提供するのみで、複数が一致した場合にどれを実際に 実行するかは呼び出し側（UI層）に委ねている。以下の「解決規則」自体は未解決のまま。
  • 同じオブジェクト内で、複数のキーが同じ with にマッチしてしまった場合の解決規則（優先順位、条件の排他性を 著者の責任に委ねるだけで良いか、など。2 節参照）
  • combinations（斧で木を切る、など）を、actor の装備スロットを経由したパス参照（例: actor.equip.tool）を 使う actions の条件・効果として書き換えられないか
  • with で、複数の trait / 条件の組み合わせ（AND条件）を指定する必要があるか
  • 対称的な組み合わせ（3 節）で両側に同じ内容を書く冗長さを、将来的に軽減する余地を残すか
  • 同一 id の定義がファイル間で重複した場合の、新しいキーの追記のマージセマンティクス （GameElementDefinition.md 3.3 節との関連）
  • showMenu の値が always 以外に増える場合の具体的な用途・記法
  • ドラッグ中に他のカードをハイライトする際、全カードに対して combinations の conditions を評価するコストを どう抑えるか
  • 素材側が「どの道具で操作されたか」を細かく区別したい場合、道具ごとに別々のキーを書き分ける以外の効率的な記法が 必要か
