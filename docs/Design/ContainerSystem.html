ContainerSystem

コンテナの容量と重さ

概要

本ドキュメントは、GameElementDefinition.md 7.1節に残っていた「slot に容量・サイズ制限を持たせるか」という未決事項について、 サイズによる容量制限と、重さの伝播・行動力への影響という2つの観点から検討した結果をまとめたものです。

RecipeSystem.md の accepts（型・個数の制約）とは独立した、合計サイズによる制約、および重さという 別の観点を扱います。

本ドキュメントは検討結果であり、確定仕様書ではありません。未決事項は 6 節に整理しています。

1. サイズ・容量

各コンテナ（スロット）は独自の容量を持ち、すべてのアイテムはサイズを持ちます。サイズの合計が容量を超えない範囲でのみ 格納できます。

slots:
  inventory:
    capacity: 50

props:
  size:
    value: 3

move_to_slot（7.1節の唯一の汎用操作）が、「現在の中身のサイズ合計＋新しいアイテムのサイズ ≤ capacity」を検証し、 超える場合は移動を拒否します。

accepts（RecipeSystem.md 5節、型・個数の制約）とは独立した仕組みであり、1つの slot が両方を同時に持つことも 妨げません（例: 型を制限しつつ合計サイズも制限する、特殊な収納）。

2. 重さの伝播: derivedではなくmove_to_slotの副作用として扱う

サイズとは別に、アイテムは重さの概念を持ちます。コンテナ自体の重さは、中に入っているアイテムの重さの影響を受けます （バックパックの中身が増えれば、バックパック自体が重くなる）。

一見すると計算式（derived、GameElementDefinition.md 6.5節で採否未確定としているもの）が必要に思えますが、 これは既存の move_to_slot の性質を使い、計算式なしで実現できます。move_to_slot は既にスロット配列と逆引きキャッシュの 整合性を保証する唯一の場所として定義されています。ここに、「アイテムがスロットに出入りするたびに、そのアイテムの 現在の実効重量を、スロットの持ち主の weight プロパティへ加算・減算する」という、もう1つの不変条件を持たせます。

  • カートに石を入れる → move_to_slot が、石の重量をカートの weight へ加算する
  • 石をカートから出す → 同じ量を減算する

これは温度・湿度（ClimateSystem.md）で採用した「毎 tick 加減算する」方式とは異なり、スロット移動という離散イベントの 度に1回だけ加減算する方式です。新しいトリガー体系を追加するのではなく、既存の move_to_slot という唯一の操作が持つ 不変条件を1つ増やすだけであり、「値は独立変数への加減算によって駆動される」という数値積分の哲学から外れていません。

入れ子（アイテム→バッグ→バックパック→装備）も、この仕組みが再帰的に働くことで自然にカスケードします。バッグ自身の weight は、バッグの中身の出入りに応じて常に最新化されているため、バッグをバックパックに入れる瞬間は「バッグの現在の weight」を1回伝播させるだけで済み、中身を毎回覗き直す必要はありません。

3. 伝播率（weight_rate）: 「身につけると軽い」と「コンテナ自体の軽減能力」を1つの仕組みに統一

「ほとんどのアイテムは手荷物より身につける方が軽い」という現象と、「台車は中身が増えても重さが増えにくい」という現象は、 同じ仕組みで説明できます。スロットごとに「伝播率（weight_rate）」を持たせます。

object_defs:
  character:
    slots:
      equip:
        weight_rate: 0.5    # 身につけると重さが半分に感じる
      inventory:
        weight_rate: 1.0    # 手荷物はそのまま

  cart:
    props:
      weight:
        value: 20           # 台車自体の重さ
    slots:
      storage:
        weight_rate: 0.1    # 中身の重さの10%しか伝播しない
  • 「身につけると軽い」→ character.equip の weight_rate を inventory より低くする
  • 「台車は中身が増えても重くなりにくい」→ cart.storage の weight_rate を低くする

どちらも「スロット側が持つ、伝播時の割引率」という同一の概念であり、コンテナごとに専用の「軽減能力」という別概念を 作る必要はありません。バックパックを背負うと軽く感じる、という現象も、character.equip と backpack.storage 両方の weight_rate が効いた結果として自然に説明できます。

4. 行動力への影響

キャラクターの合計重量（各スロットからの伝播の総和）を weight のようなプロパティとして持たせ、既存の stages （GameElementDefinition.md 6.4節）で「軽い/やや重い/かなり重い/動けない」のような段階に分けます。あとは GameElementDefinition.md 8.1節の move アクションの例（world.light_stage で移動可否を制御した例）と同じパターンで、 actor の重量段階を条件・効果に使えます。

actions:
  move:
    conditions:
      - { path: actor.encumbrance_stage, op: neq, value: immobile }

移動時間の延長・疲労の増加も、それぞれの段階の effects で modify/add すればよく、新しい仕組みは不要です。 上限を設けない、という要件も、stages の最下段・最上段が min のみを指定する半開区間である以上、自然に満たされます。

5. なぜプレイヤーが持っている場合にのみ影響するか

重さの伝播自体は、どのオブジェクトの slots でも同じように起きます。特別扱いは必要ありません。「行動力に影響する」のは キャラクター自身の weight を読む stages/actions だけなので、地面に置かれた宝箱の重さが誰の行動力にも影響しないのは 自然な帰結です。

6. 未決事項・今後の検討課題

  • 既に中に入っているアイテムの重さが後から変化した場合（例: 松明の燃料が減って軽くなる）、move_to_slot は出入りの 瞬間にしか発火しないため、入れっぱなしのまま重さが変わってもコンテナ側の weight は自動的に追従しない。これを 許容する仕様上の割り切りとするか、何らかの再伝播の仕組みを別途用意するか
  • size/capacity が単純なスカラー数値で十分か、それとも将来的に形状（テトリス型インベントリ等）を考慮したいか
  • weight_rate が 1.0 を超える（かえって重く感じる）ケースを想定する必要があるか
  • size/capacity/weight_rate の具体的な数値・単位（本書のサンプルはあくまで説明用）
  • 重さの伝播（2節）と RecipeSystem.md の accepts から自動生成される combinations（材料投入）が、同じ move_to_slot イベントに対してどちらも副作用として発火する場合の実行順序

7. 参考: 既存プロジェクト方針との整合性

  • capacity（本書）と accepts（RecipeSystem.md 5節）は、いずれも slots への独立した拡張であり、GameElementDefinition.md 7.1節に残る「slot の容量・型制約」という同じ未決事項の、異なる側面への回答です。
  • 重さの伝播を move_to_slot の不変条件として扱う設計は、7.1節が既に「スロット間の移動は move_to_slot という単一の 汎用操作に集約する」と定めていることの延長であり、新しいトリガー体系を追加していません。
  • 重さの伝播に derived を使わなかった判断は、ClimateSystem.md・PickSystem.md に続く、独立変数への加減算のみで 現象を表現するという方針の一貫した適用です。ただし本書の重さは、温度・湿度のような「時間経過にともなう蓄積」ではなく 「現在の中身の構成」を反映する値であり、加減算のタイミングが tick 単位ではなくスロット移動イベント単位である点が異なります。
  • 行動力への影響（4節）は、GameElementDefinition.md 8.1節の light_stage/move の例と同じ、stages による状態決定パターンの 再利用です。
