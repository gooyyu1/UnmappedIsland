TerrainGeneration

地形生成システム設計

概要

本ドキュメントは、島の地形生成システムに関する検討結果をまとめたものです。 GameElementDefinition.md が掲げる「ハードコードしない」「汎用エンジンに任せる」「ファイル追加だけで拡張できる」という 設計方針（2.4 節）に準拠し、以下を目的とします。

  • シード値ありのランダム生成で無人島の地形を生成する
  • 地形・構造物はすべて YAML でオブジェクト定義し、MOD 作成者がコードに触れずに拡張できる状態を保つ

本ドキュメントは検討結果であり、確定仕様書ではありません。実装・詳細スキーマ化は本書を土台に別途進めます。 「未決事項」は 6 節に選択肢のみを提示する形で明記しています。

1. 用語定義

これまでの議論で用語のブレを解消済みです。以降のドキュメント・コードでは以下の呼称に統一します。

用語 説明
Axis（軸） 標高・湿度など、地点が持つ連続値パラメータの 1 次元。ジェネレータ（ノイズ/距離場/散布など）によって値が決まる
Site（地点） 座標と軸ベクトルのみを持つ、LocationType 未確定の中間状態のノード
環境（Environment） 軸ベクトルが表す性質そのもの。データ構造として実体化はしない（クラスや YAML 定義を持たない）。Site 同士の軸ベクトルの近さという関係性としてのみ存在する概念
LocationType 「草原」「洞窟」など、配置の定義（YAML テンプレート）。旧称 TerrainType/RoomType を統一した語。プレイヤーには見えない設計者側の語彙
Location（土地） 実際に生成・命名された実体。旧称 Land/Room を統一した語。「東の草原」のようにゲーム内で識別可能な名前を持つ。パスネットワークのノードでもある
Structure（構造物） 内部に子 Location のグラフを持つ、特殊な Location。「島の中に埋め込まれた入れ子の島」として扱う
Coordinate/Position 単なる座標値。Location が持つプロパティの一つ。Location という語自体は「意味を持つ場所の単位」を指し、座標そのものとは区別する

2. 生成パイプライン全体像

[Axis定義 (YAML)]
   ↓ ジェネレータ(ノイズ/距離場/散布/合成)が値をサンプリング
[Site群] (座標 + 軸ベクトル)
   ↓ LocationType の axis_preferences と最近傍マッチング
   ↓ (フォールバック処理 / バランス調整パス)
[Location群] (LocationType確定・命名済み)
   ↓ Delaunay三角形分割 → MSTで間引き → 交差なしパス確定
[Locationネットワーク] (距離・移動難易度つきパスで接続)
   ↓ StructureType の allowed_structures を参照し、対象Locationに構造物を配置
   ↓ 構造物内部は同じパイプラインを縮小スケールで再帰的に実行
[Structure] (内部に子Locationグラフを持つLocation)

重要な設計方針: 島の生成と構造物内部の生成は同一の生成ロジックを共有します。両者の違いは「生成スコープ（scale）ごとのパラメータプリセット」の違いとして表現し、ロジックを分岐させません（4.7 節参照）。

3. 各ステップの詳細仕様

3.1 Axis定義とジェネレータプリミティブ

Axis は以下の汎用プリミティブの組み合わせで値を生成します。MOD 作成者は新しいプリミティブをコードで追加せずとも、既存プリミティブの組み合わせで大半の要件を満たせるようにします。

  • layered_noise: パーリン/シンプレックス等のノイズ。オクターブ・周波数指定可
  • distance_field: 島の縁・中心・任意参照点からの距離場
  • blob_scatter: ランダムなエピセンター＋減衰。局所的に濃い領域（例: 汚染地帯）を作るのに使う
  • modifiers: 他の軸との掛け算・反転・クランプなどの合成演算

設計上の注意: 軸の種類・数はハードコードしません。Axis 定義自体が YAML で完結し、LocationType 側は「言及した軸だけ気にする」設計にすることで、軸の増減に対して LocationType 定義が壊れないようにします（3.2 節参照）。

- id: pollution
  range: [0.0, 1.0]
  generator:
    type: blob_scatter
    sources: { count: [1, 3], seed_offset: 1337 }
    falloff: inverse_square

3.2 LocationTypeマッチング（軸ベース）

各 LocationType は、軸空間上の「理想点＋許容範囲」として定義します。

- id: inland_jungle
  applicable_scopes: [island]        # 3.7節参照。適用範囲を制限できる
  axis_preferences:
    elevation:  { ideal: 0.1, tolerance: 0.3, weight: 1.0 }
    humidity:   { ideal: 0.8, tolerance: 0.2, weight: 1.5 }
    # 言及しない軸は自動的に「無関心」= weight 0 として扱う
  hard_limits:
    elevation: { max: 0.6 }          # 絶対的な除外条件
  • マッチングは正規化した重み付きユークリッド距離による最近傍探索です。
  • GameElementDefinition.md が定義する「範囲による状態の決定」パターン（2.3 節。空腹度→満腹/空腹/飢え、等）と同一の仕組みであり、新規の専用エンジンではなく既存の汎用パターンの転用として実装します。

3.3 マッチできない場合の処理（フォールバック）

3 段構えで対応します。

  1. フォールバック LocationType: is_fallback: true、全軸無関心、priority 最低。候補がない場合はこれが選ばれる
  2. 段階的緩和: tolerance を超える場合、最も合っていない軸から順に無視して再探索
  3. フォールバックの機能化: 特定の軸（例: 汚染度）にのみ強くマッチする専用 LocationType を MOD 側で追加すれば、フォールバック機構が自動的にその領域へ誘導する。「マッチしない領域」は不具合ではなく「新しい LocationType を定義すべき場所」というシグナルとして扱う

3.4 出現数について（方針転換の記録）

個別の LocationType ごとの出現数レンジ管理は不要です。 軸ベースの最近傍マッチングは各 Site が独立に判定されるため、個数の厳密な制御には本質的に向いていません。

代わりに重視するのは島全体のバランスです。例:

  • 乾燥地帯・湿地が最低 1 つ以上存在する
  • 山が 1 つ以上存在する

これは個々の LocationType の出現数ではなく、「特定の軸領域が島内に一定以上出現しているか」という軸空間のカバレッジの問題として扱います。具体的な保証アルゴリズム（軸値の分布を事後チェックし、不足していれば blob_scatter のパラメータを調整して再生成する、など）は未決事項とします（6 節参照）。

3.5 パスネットワーク生成（交差なしグラフ）

  • ノード（Site→Location）配置は Poisson-disk sampling 等で島内に散布します。
  • 全ノードに対して Delaunay 三角形分割 を実施します（数学的に辺の交差が起きない性質を利用）。
  • 間引きは以下の 2 段階で行います。
    1. 最小全域木（MST）は必ず残す（到達性の保証）
    2. MST 以外の辺は距離・軸の近さに応じたスコアで一部復活させ、近道・分岐を作る
  • 「交差する代わりに新しい地区を生成したい」の実装方針: 直接つなぎたいが交差してしまう 2 ノードがある場合、交差点付近に新規 Location（候補: 小規模な waypoint 用途の地区）を挿入し、局所的に Delaunay 再分割を行います。これによりグラフは常に平面性を保ちます。
  • 各エッジには distance（ユークリッド距離）と difficulty（両端 LocationType の移動コスト平均×距離補正）を持たせます。

3.6 命名処理

  1. 島の重心を計算する
  2. 各 Location の重心からの方角（8 方位）を算出する
  3. "{方角} + LocationType.display_name" で仮称を生成する（例: 「東の草原」）
  4. 重複時は LocationType の name_pool から固有名詞を採用、またはフォールバック接尾辞（第一/第二等）で区別する
  5. 印象的な地形（山頂、極端環境ノードなど）は name_pool の固有名詞を優先する

3.7 構造物（Structure）生成

  • Structure は「内部に子 Location グラフを持つ、特殊な Location」として扱います。
  • 内部生成は 3.1〜3.6 の同一パイプラインを、生成スコープ（scope）ごとのパラメータプリセットを差し替えて再帰的に実行します。
# generation_scope.yaml (案)
- id: island
  site_count_range: { min: 40, max: 80 }
  axes: [elevation, humidity, coastal_distance, pollution]
  branch_factor: { min: 1, max: 3 }

- id: structure_interior
  site_count_range: { min: 3, max: 7 }     # 小規模・迷いにくさの要件をここで表現
  axes: [dampness, light_level, decay]      # 島とは異なる軸セットでよい
  branch_factor: { min: 0, max: 1 }
  dead_end_ratio_max: 0.3
  • LocationType 側の applicable_scopes タグにより、「洞窟の中に草原は出ない」等の制約を自然に表現できます。
  • 両方のスコープに効く LocationType（例: 「地底湖」を島にも構造物内部にも出す）も定義可能です。

4. これまでに合意した設計原則のまとめ

  • 地形定義（LocationType）と実体（Location）は必ず呼び分けます。裸の名詞＝実体、〜Type サフィックス＝定義、という命名規則を全概念で徹底します。
  • 「似た環境が隣り合う」は、隣接しやすさテーブル（affinity）ではなく、軸の空間的連続性そのものによって自然に実現します（affinity 方式は不採用）。
  • 稀な極端環境差は、extreme_weight のような人為的パラメータではなく、高周波の “weirdness 軸” や急峻な勾配によって自然発生させることを検討します。
  • ハードコーディングを避けるため、Axis の生成ロジックは汎用プリミティブ（ノイズ/距離場/散布/合成）の組み合わせに還元します。
  • 島全体の生成と構造物内部の生成は、同一の生成ロジックをスコープ別パラメータで使い分けます。ロジック自体は分岐させません。

5. 生成パイプラインの前段・後段（範囲外の確認事項）

本書が扱うのは Axis 定義から Structure 生成までの地形生成パイプラインそのものです。生成された Location/Structure に対する アイテム配置・イベント配置・プレイヤーの探索状態管理などは範囲外とし、別途検討します。

6. 未決事項・今後の検討課題

  • 軸空間のカバレッジ保証アルゴリズム: 「山が 1 つ以上」「乾燥地帯が 1 つ以上」等のバランス要件を、具体的にどう保証するか（事後チェック＋再生成 / 生成前に軸のヒストグラムを調整 等）
  • weirdness 軸の具体設計: 稀な異常環境をどの程度の頻度・強度で発生させるか
  • Axis の計算コスト: 軸の数が増えた際のサンプリングコスト、キャッシュ戦略
  • 合成演算（modifiers）の YAML 文法: どこまで柔軟な合成を許容するか（単純な四則演算のみか、条件分岐を許すか等）
  • 構造物内部の部屋の意味論: Location として統一したことで、内部の各部屋にも固有名（「祭壇の間」等）を与える運用ルールの詳細化
  • YAML 定義のマージ・上書き規則: 同一 ID が複数ファイルに存在する場合の挙動（GameElementDefinition.md 3.4 節で「別途仕様書で定義する」とされている未着手事項）
  • 具体的なスキーマ・サンプル YAML の作成: 本書はまだ概念設計の段階であり、実際のフィールド名・型・ディレクトリ配置などのスキーマは未作成

7. 参考: 既存プロジェクト方針との整合性

Documents/Design/GameElementDefinition.md に示されている以下の原則と、本設計は整合しています。

  • すべての概念を YAML で定義し、複数ファイル分割・MOD 追加は別ディレクトリへのファイル追加のみで実現する（2.1 節）
  • 「ロケーション」は既に正式概念（地区・建築物・洞窟等）として文書内に例示されており（2.2 節）、本書の Location/LocationType はこれを具体化したものである
  • 「範囲（値域）による状態の決定」という汎用パターン（2.3 節。空腹度→満腹/空腹/飢え、等）が既に採用されており、LocationType の軸マッチングはこのパターンの応用として実装すべきである

なお、GameElementDefinition.md 10 節では地形を object_defs の一種として表現し、terrain_candidate trait の generation_weight を例示していますが、これは「trait で必須 props を宣言する」という YAML の一般的な仕組みを示すための簡易例であり、 実際の LocationType 選定アルゴリズムは本書 3.2 節の軸ベース最近傍マッチングに置き換わります。
