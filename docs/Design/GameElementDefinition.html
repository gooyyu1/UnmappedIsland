GameElementDefinition

ゲーム内要素の定義方式

1. 位置づけ

本ドキュメントは、ゲーム内要素を YAML で記述するための設計を、以下の 3 つの観点からの議論を統合して整理したものです。

  • 全体方針（何を、なぜ YAML で表現するか）
  • ファイル全体・型定義まわりの具体的な YAML 構造（WorldCodex / object_defs / traits / props）
  • オブジェクト同士の影響関係を表す効果システム（親子スロット、装備の排他制御、effects）

いずれも粒度の異なる議論の積み重ねであるため、確定した仕様と未決定の論点が混在しています。 「確定事項」は議論済みの決定として記載しますが、「未決事項」は結論が出ていないため、各節および末尾の 12 節に TODO: 未決定 として選択肢のみを提示します。本ドキュメントは検討結果であり、確定仕様書ではありません。

文中に登場する防具・細菌感染・耐久値などの具体例は、記法を説明するためのサンプルであり、 実際のゲームバランスや病気システムそのものを規定するものではありません。

2. 設計思想

2.1 YAML定義の全体方針

ゲーム内に登場するほぼすべての概念は YAML 形式 で定義します。 定義ファイルは複数のファイルに分割して配置できるため、ゲーム本体が提供するデフォルト定義とは別に、 ユーザー（MOD 作成者）がファイルを追加するだけで新しい要素を導入したり、既存の要素を上書き・変更したりすることができます。

管理する概念は、型定義（ObjectDef）・実行時インスタンス（ObjectInstance）・全体（WorldCodex）の 3 つです。 このうち ObjectInstance 側の詳細フォーマットは未着手であり、本ドキュメントは主に ObjectDef 側の構造を扱います。

2.2 定義できる概念の種別

世界レベルの概念

ゲーム内時刻・日付・季節・天候など、世界全体に影響する概念を定義します。

  • 時刻の刻み方（1 tick あたりの経過時間）
  • 昼・夜の時刻範囲
  • 季節の日付範囲
  • 天候の種類と遷移ルール

ロケーション

プレイヤーが探索できる場所を定義します。

  • 地区（森、海岸、廃村など）
  • 建築物・施設
  • 洞窟や特殊地形

プレイヤー・キャラクターパラメータ

キャラクターが保持すべき数値パラメータを定義します。

  • 空腹度、喉の乾き、眠気
  • 体温、血液量
  • 幸福度、傷みなど

アイテム

プレイヤーが入手・使用・製作できる物を定義します。

  • 設備・道具
  • 資材・食料
  • 薬品・素材など

アイテム属性

アイテムが持つ性質（燃焼性、腐敗速度、重量など）を定義します。属性はアイテムに付与する形で参照され、アクションの結果に影響します。

アイテムアクション

アイテムに対して実行できる操作（食べる、使う、加工するなど）を定義します。アクションは実行可否の条件と、実行結果（effects）を 1つの定義としてまとめて持ちます（詳細は 8 節、ActionSystem.md）。

2.3 統一的な実行エンジン

可能な限り、すべての概念は共通の枠組みで表現され、単一の汎用エンジンによって実行されます。 「特定の概念専用のロジック」を増やすことを避け、拡張性と MOD 互換性を確保します。

tick 駆動による変化

ゲーム内のあらゆる「時間経過にともなう変化」は tick を基本単位とする同一メカニズムで実現します。

現象 仕組み
ゲーム内時刻の進行 1 tick ごとに「時刻」パラメータを加算
空腹度の増加 1 tick ごとに「空腹度」パラメータを加算
傷の出血による血液量の減少 1 tick ごとに「血液量」パラメータを減算
部屋の快適度によるキャラクターの気分の変動 1 tick ごとに「幸福度」パラメータを加減算

範囲による状態の決定

パラメータの値域から状態を決定する仕組みも共通化します。

現象 仕組み
時刻の範囲で昼夜を決定 時刻パラメータの値域で「昼」「夜」を割り当て
日付の範囲で季節を決定 日付パラメータの値域で「春」「夏」「秋」「冬」を割り当て
空腹度の範囲で飢えや満腹の状態を決定 空腹度パラメータの値域で「満腹」「空腹」「飢え」を割り当て

このように、単純なパラメータとその値域の定義だけで、世界の挙動もキャラクターの状態も同一の手法で記述できます。 この「値域による状態決定」を型定義レベルで一般化したものが、6.4 節で述べる stages（段階）です。

2.4 設計上の方針

  • 「ハードコードしない」：コードに定数や挙動を埋め込まず、YAML 定義で制御できるようにする
  • 「汎用エンジンに任せる」：特定概念向けの例外処理をエンジン側に追加しない
  • 「ファイル追加だけで拡張できる」：MOD 作成者がコードに触れることなく新要素を追加できる

3. ファイル全体構造とロード

3.1 WorldCodexとルート構造

ファイル全体が 1 つの WorldCodex を表します。専用のルートキーは置きません（ファイル自体が WorldCodex です）。

3.2 命名規則

  • 構造キーワード（props、traits、actions など）・型名・プロパティ名は、すべて snake_case で統一します。非プログラマも編集するため、camelCase との混在を避けます。

  • id（型名・プロパティ名・トレイト名・アクション名・スロット名に共通）の命名規則は次の正規表現に従います。

    ^[a-z][a-z0-9_]*$

3.3 パーサ要件（重複キーの扱い）

YAML 仕様上マッピングキーは一意であるべきですが、多くのパーサ実装（素の PyYAML 等）は重複キーをエラーにせず後勝ちで上書きします。 これは MOD 由来の記述ミスなどを静かに握りつぶす危険があるため、以下の二重の安全策を仕様として明記します。

  • ローダーは重複キー検出でエラーを出す厳格モードを使用する
  • 加えて、ロード後に別途バリデーションステップを設ける

3.4 YAML定義の分割とロード

YAML ファイルは所定のディレクトリ以下に置かれたすべてのファイルが自動的に読み込まれます。 ゲーム本体の定義は data/ などの標準ディレクトリに、MOD の定義は別ディレクトリに配置することで、双方を混在させずに管理できます。

同じ ID を持つ定義が複数のファイルに存在する場合のマージ・上書き規則は別途仕様書で定義します。

4. object_defs（型定義）

型定義は object_defs の下に、識別子をキーとして表現します（値ではありません）。

object_defs:
  stone:
    props:
      weight:
        value: 10

5. traits（mixin）

記述量削減のため、複数の object_def で共有するプロパティ群を traits として定義できます。

  • 多重継承は禁止: trait が他の trait を参照することはできません（1 階層のみ）。
  • プロパティ衝突はエラー: 複数の trait が同名プロパティを宣言していた場合、暗黙の優先順位で解決せずエラーとします（安全側に倒す）。
  • 値の扱い:
    • trait 側で value を持たないプロパティ = object_def 側で値を与えることが必須
    • trait 側で value を持つプロパティ = デフォルト値。object_def 側で省略可、上書きも可
    • object_def 側で一部属性（例: value）だけを上書きした場合、残りの属性（unit など）は trait 側の値を引き継ぐ
traits:
  perishable:
    props:
      shelf_life:      # valueなし→継承先で必須
        unit: days
      decay_rate:
        value: {min: 1, max: 3}   # valueあり→デフォルト値(範囲値は6.2節参照)

object_defs:
  apple:
    traits: [perishable]
    props:
      shelf_life:
        value: 7

アクション・効果も trait 経由でまとめて配布できます（7〜8 節参照）。例えば eatable trait が eat アクションと関連 props をセットで提供する、といった使い方を想定します。

6. props（プロパティ）

6.1 固定値

最も単純な形は、value に固定値を 1 つ持つだけの形です（4 節の stone.props.weight を参照）。

6.2 範囲値（ランダム再ロール）

固定値の代わりに {min, max} の範囲を value に持たせ、毎 tick ごとに範囲内でランダムに値を再ロールできる仕組みを持ちます。 腐敗速度・天候の持続時間など、複数の用途で共有する汎用プリミティブとして設計します。

props:
  decay_rate:
    value: {min: 1, max: 3}   # 毎tick、この範囲でロールした量を減算する

6.3 overflow（周回・上限のあるプロパティ）

時刻のように上限に達したら折り返す（あるいは繰り上げる）プロパティのため、range と on_overflow を持たせます。 value の {min, max}（6.2 節、毎 tick 再ロールする範囲）とは異なる仕組みである点に注意してください。 range は値が取りうる上下限、on_overflow は上限到達時の挙動を表します。

props:
  minute_of_day:
    value: 0
    range: {min: 0, max: 1439}
    on_overflow: {mode: wrap, carry_to: day}   # 1440でdayに+1して0に戻る
  sequence:
    value: 0
    on_overflow: {mode: none}                   # 上限なし

6.4 stages（段階）

数値パラメータ（プロパティ）は stages を持つことができ、現在値がどの段階にあるかによって、有効になる effects（8 節）が切り替わります。

  • 段階の区間定義は min（下限）のみを指定します。上限は明示せず、次の段階の min によって自動的に決まります（半開区間、[min, 次の min より手前]）。この方式により、区間の隙間や重複が構造的に発生しません。
  • 最下段の段階は min: null（または省略）とし、それより下の残り全ての値を拾います。
  • 現在値に基づいて常に一意に段階を決定します。ヒステリシス（上昇時・下降時で閾値をずらす仕組み）は採用しません。プレイヤーが常に現在の状態から結果を逆算できるようにし、状況分析をしやすくするための設計意図です。
  • ステージの effects はレベルトリガー（そのステージにいる間ずっと有効）を基本とします。ステージ切替の瞬間だけ発火する仕組み（edge-triggered、例えば on_enter 的なもの）は将来検討事項として保留中です。
properties:
  durability:
    value: 100
    stages:
      - name: intact
        min: 1
      - name: broken
        min: null

具体的な effects と組み合わせた例は 11.2 節を参照してください。

7. オブジェクトの親子関係とスロット

7.1 親子関係とスロットの基本構造

  • オブジェクト間の所属関係はツリー構造で表現します。1 つの子オブジェクトは必ず 1 つの親に属し、複数の親を持つ関係は採用しません。
  • 親オブジェクトは複数の「スロット」（配列）を持ちます。例: inventory（所持品）、equip（装備）、fixtures（部屋の備え付け設備）、injuries（怪我）。
  • 子オブジェクトは held_in のような逆引き用のキャッシュフィールドを持ちますが、正の情報源はあくまで親側のスロット配列です。
  • スロット間の移動は move_to_slot という単一の汎用操作に集約します。親側スロット配列と子側の逆引きキャッシュの整合性は、この操作の中でのみ保証します。個別の YAML 定義側が両者を直接書き換えることは想定しません。
  • スロットに「種類タグ（kind）」は設けません。equip はそれ自体が唯一の装備用スロットであり、スロット名を直接参照します。スロットの意味をタグで抽象化するよりも、名前をそのまま参照する方が単純で、現時点で複数の装備スロットを区別する必要も生じていないためです。
object_defs:
  character:
    props:
      hp:
        value: 100
    slots:
      equip: {}
      inventory: {}
      injuries: {}

7.2 装備の排他制御（layer / covers）

  • equip スロットには複数のアイテムを同時に格納できます。排他制御はスロットの数ではなく、アイテムが持つ属性によって行います。
  • アイテムは covers（覆う部位のタグの配列。例: [torso]、[legs, waist]）と layer（重ね着の階層。例: base、outer）を持ちます。
  • 競合判定は、同じ (covers要素, layer) の組が、既に equip スロットに入っている他アイテムと重複する場合に装備不可とします。
    • 例: シャツ（torso, base）とジャケット（torso, outer）は共存可能です（layer が異なるため）。
    • 例: シャツ（torso, base）とつなぎ（legs/waist/torso, base）は共存不可です（torso, base が競合するため）。
  • 競合解決は「ブロック型」を採用します。競合するアイテムがある場合、プレイヤーが先に既存の装備を外す必要があります（自動退避は採用しません）。装備の入れ替えを明示的な操作として要求することで、プレイヤーが装備状態を把握しないまま暗黙に変更されることを避けるためです。

8. アクションと効果（actions / effects）

8.1 actions（メニュー型の宣言的操作）

  • アクション（eat、move など）は、条件（conditions）と実行結果（effects）を1つの定義としてまとめて持ちます。かつては 「条件だけを持つ受動的な宣言」と「on: <アクション名> で別途発火する効果」を分離していましたが、1つの操作の挙動が複数箇所に 分散して読みにくくなるため、この分離は廃止しました（詳細な経緯・カード間の相互作用（ドラッグ操作）は ActionSystem.md を参照）。
  • アクションは object_defs および traits の中に配置します（トップレベル独立キーにはしません）。これにより、traits 経由でアクションもまとめて配布できます。
  • showMenu でメニューへの表示方法を制御します（現時点では always のみ。将来的にボタン以外の起動方法が増えた場合に備え、列挙可能な形にしています。詳細は ActionSystem.md）。
  • effects は、このアクションが実行された瞬間に 1 回だけ適用されます（8.3 節の add の適用頻度ルール参照）。
traits:
  eatable:
    actions:
      eat:
        showMenu: always
        conditions:
          - {path: actor.satiety, op: lt, value: max}
        effects:
          actor:
            add:
              satiety: 10
          self:
            lifecycle:
              destroy: true

明るさで移動可否を制御する例です。

object_defs:
  world:
    singleton: true
    props:
      light_stage:
        value: bright   # 6.5節の議論次第でderivedになる可能性あり(未確定)

  character:
    props:
      hp:
        value: 100
    actions:
      move:
        showMenu: always
        conditions:
          - {path: world.light_stage, op: in, value: [bright, dim]}

条件式は {path, op, value} の形を取ります。path は参照ルートから始まるドット区切りのパスです。 現時点で定義されている参照ルートは self（このアクションを宣言したオブジェクト自身）・parent（その親）・ actor（このアクションを実行しているプレイヤーキャラクター）・world（環境シングルトン、9 節参照）の 4 つです。 self/parent/actor は 8.2 節の effects のキーと同じ語彙・同じ意味で使います。

actor はすべてのアクションに暗黙的に存在し、常にプレイヤーキャラクターを指します。 eat の例のように、 アクションを持つオブジェクト（食べ物）自身は消費される側（self）であり、実際に効果を受け取るのは食べ物を保持している 容器（parent）ではなく、行動主体である actor です。parent はあくまで木構造上の直接の格納先を指すものであり、 actor とは独立した参照です（例えば地面に置かれた食べ物を直接食べる場合、parent はキャラクターではなく地面のロケーションになりますが、 actor は変わらずプレイヤーキャラクターを指します）。

カードを別のカードにドラッグする操作（例:「斧で木を切る」）は、actions ではなく combinations という別の仕組みで表現します。 一方、攻撃のように、行動主体（actor）以外に登場する参加者が「メニューを開いたカードそのもの」（例: 攻撃対象の敵カード） だけで済む場合は、ドラッグ操作を使わず actions（メニュー型）で表現します（詳細は ActionSystem.md）。

8.2 effectsの基本構造

効果は次の 3 つのレベルで定義でき、いずれも同一の記法・実行原理を用います。

  1. オブジェクトレベル（例: 防具そのものが持つ複数の恩恵の束）
  2. プロパティレベル（例: アイテムの重量プロパティが持ち主の負荷に寄与する）
  3. プロパティのステージレベル（例: 耐久値の残量帯によって防御効果が変わる。6.4 節参照）

effects は、効果の対象（self/parent/child/actor）を識別子とするキーとして表現する辞書型です。 現時点で定義するのは self（自分自身）・parent（親）・child（子）・actor（このアクションを実行しているプレイヤーキャラクター、8.1 節参照）の 4 つです。 ancestor（先祖）・sibling（兄弟）・descendant（子孫）は、必要性が生じた時点で改めて検討します。

effects:
  parent:
    when: equip
    modify:
      defense: 5
      speed: 3
      accuracy: 2
  • when: <スロット名> … そのスロットに入っている間、継続的に有効（レベルトリガー）
  • when を省略した場合は「常時（無条件）」を意味します。

when は各対象（self/parent など）のキーの中に書きます。1 つの effects の中で、対象ごとに異なる when を 持たせることもできます（例: parent は when: equip の間だけ、self は常時、など）。トリガーを effects 全体の 兄弟キーに引き上げる案も検討しましたが、同じオブジェクトが複数の異なるトリガーを将来持つ可能性を残すため、 対象ごとに when を持たせる現在の形を採用しています。

特定のアクションが実行された瞬間にだけ発火する効果は、on: <アクション名> のような独立したトリガーとしては表現しません。 アクション自身の effects（8.1 節）、またはカード間の組み合わせを表す combinations の effects（ActionSystem.md）として、 アクションの定義そのものに直接書きます。

8.3 効果の種別（modify / add / lifecycle）

3 種類の効果種別を区別します。この区別は「値の可逆性」と「適用頻度」という 2 軸に基づきます。

  • modify: 条件が真である間だけ計算式に寄与する、可逆的な修飾子です。条件が偽になった瞬間に寄与は自動的に消えます。値そのものを書き換えるのではなく、実効値を都度導出するための入力として扱います。
    • 同一プロパティに対する複数の modify は単純加算で合成します。
    • 合成結果の上限・下限（min/max）はプロパティ（スタット）側の定義に持たせます。
  • add: 対象プロパティの実体値を直接、不可逆に加減算します。
    • actions/combinations（8.1 節、ActionSystem.md）の effects に書かれた add は、そのアクション（またはカードの組み合わせ）が 実行された瞬間に一度だけ加算します。
    • when（レベルトリガー）配下、および無条件（常時）の add は、条件が真である tick 毎に加算します（例: 出血中の血液量減少、耐久値の毎 tick 減少）。
    • 「アクション実行に伴う効果なら一回、when/常時なら毎 tick」という対応は確定した仕様です。
  • lifecycle: プロパティの値ではなく、オブジェクトそのものの存在を操作します（例: destroy でオブジェクトを消滅させる、spawn で新規オブジェクトを生成する）。modify/add が値操作であるのに対し、lifecycle は存在操作を担う第三のカテゴリとして区別します。
    • lifecycle は modify/add と同様に、複数の動詞を同時に持てる辞書形式です（例: {spawn: {...}, destroy: true}）。同じ対象に対して「新しいオブジェクトを生成しつつ、自分自身は消滅させる」という組み合わせを1つのエントリで表現できます。

modify/add/lifecycle はいずれも、対象（self/parent など）ひとつに対して「何が起きるか」を確定的に記述するものです。 複数の結果から重み付き確率で1つを選ぶ必要がある場合は、effects を書く代わりに pick を使います。pick は effects と同じ場所に書く代替のキーであり、各候補が自分自身の effects（対象ごとの辞書）を丸ごと持てます。 これにより「確率で分岐した結果、候補ごとに異なる対象へ効果が及ぶ」ケースも表現できます。詳細は PickSystem.md を参照してください。

8.4 カード間の相互作用（menu / combinations）

プレイヤーがカードに対して行う操作は、大きく2種類に分かれます。

  • メニュー型: 1枚のカードだけで完結する操作。8.1 節の actions として、条件と効果をまとめて宣言します。すべてのアクションには actor（プレイヤーキャラクター）が暗黙的に参加できるため、eat（食べ物 self ＋ actor）だけでなく、attack（攻撃対象の敵 self ＋ actor） のように、一見「2つの参加者が要る」ように見える操作も、もう一方が常に actor であればメニュー型で表現できます。
  • ドラッグ型: あるカードを、actor ではない別の特定のカードにドラッグ＆ドロップして行う操作（「斧で木を切る」など、 道具や素材そのものを指定する必要がある場合）。ドロップされた側（受け側）の combinations として宣言します。

ドラッグ型の詳細な記法（combinations の構造、ドラッグされてきたカードを指す dragged という参照、MOD 拡張性、 対称的な組み合わせの扱いなど）は ActionSystem.md に切り出しています。

以前は、アクションが成立する条件と、成立した結果として起きる効果を、on: <アクション名> という独立したトリガーを介して 別々に記述する案を検討していました。これは「アクションを宣言していない第三者も、そのアクション名を共有トリガーとして listen できる」という拡張性を狙ったものでしたが、1つの操作の挙動が複数のオブジェクトに分散して記述されることになり、 挙動を把握しにくくなる（Wiki 等での文書化がしづらい）という欠点が大きいと判断し、採用しませんでした。 この判断の詳細な経緯は ActionSystem.md に記載しています。

9. singletonオブジェクト（world）

ゲーム動作に必須ないくつかの情報（日時、天候、明るさ等）を持つ、唯一のインスタンスを想定しています。 「インスタンスが 1 つだけ存在すべき」という制約は、専用のルートキーを作るのではなく、object_defs のエントリに小さなフラグ（singleton: true）を追加することで表現します。

object_defs:
  world:
    singleton: true
    props:
      sequence:
        value: 0
        on_overflow: {mode: none}
      day:
        value: 1
        on_overflow: {mode: none}   # TODO: 未決定。年単位でのwrapについては6.3節を参照
      minute_of_day:
        value: 0
        range: {min: 0, max: 1439}
        on_overflow: {mode: wrap, carry_to: day}
      weather:
        value: clear

sequence・day・minute_of_day は 1 つの複合値にまとめず、独立した props として持ちます。 6.3 節の overflow 機構は prop 単位（range / on_overflow）で定義される仕組みであり、複合値の内側の要素それぞれに overflow 設定を紐づける手段は存在しないためです。

日時・天候はオブジェクトから直接参照されるのではなく、環境がオブジェクトに影響を与えるという位置づけです（例: 明るさによって行動可否が変わる）。 直接のプロパティ参照経路（world.xxx のようなパス表記）は 8.1 節の action 条件式で使う想定です。

10. terrain / 島生成

地形は object_defs の一種として表現しますが、島の自動生成アルゴリズムが必要とする属性群があります。 これは新しい構造を作らず、trait として必須 props を宣言することで表現します。

traits:
  terrain_candidate:
    props:
      generation_weight: {}   # valueなし→object_def側で必須

object_defs:
  forest:
    traits: [terrain_candidate]
    props:
      generation_weight:
        value: 5

「value なしプロパティ」は、5 節の trait の値継承ルールに従い、空のマッピング（{}）として表現する形に統一しています。

上記はあくまで「trait で必須 props を宣言する」という仕組みの一般例であり、実際に地形（LocationType）を どう選定するかの具体的なアルゴリズムは TerrainGeneration.md で軸ベースの最近傍マッチングとして別途定義しています。

covers: [torso]
layer: base

11.2 耐久値（プロパティのステージレベル、毎 tick 減少と消滅条件）

properties:
  durability:
    value: 100
    effects:
      self:
        add:
          durability: -1
    stages:
      - name: intact
        min: 1
      - name: broken
        min: null
        effects:
          self:
            lifecycle:
              destroy: true

11.3 細菌感染（オブジェクト化された「条件」、段階に応じた多段の影響）

感染度自体をキャラクターの生パラメータにはせず、conditions のようなスロットに格納される独立したオブジェクトとして扱います。 怪我・疲労などの「条件」は複数同時に存在しうるため、条件ごとに独立したオブジェクトとして表現する方が、キャラクター側のパラメータを増やし続けるより見通しが良いためです。

感染オブジェクトの progress プロパティが stages を持ち、段階ごとに体温・水分・意識などの他プロパティへ影響します。

condition: infection
properties:
  progress:
    stages:
      - name: none
        min: 0
      - name: mild
        min: 20
        effects:
          parent:
            add:
              temperature: 0.1
      - name: feverish
        min: 50
        effects:
          parent:
            add:
              temperature: 0.2
              hydration: -1
