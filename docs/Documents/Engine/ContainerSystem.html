コンテナの容量と重さ

概要

本ドキュメントは、slots の weight_rate（GameElementDefinition.md 7.4 節）を使って、重さの伝播・行動力への 影響をどう成立させるかをまとめたものです。capacity（合計サイズの制約、同 7.3 節）は move_to_slot が検証するだけの 単純な仕組みで、文法以上の設計判断を要しないため、本書では重さの伝播にのみ焦点を当てます。capacity/weight_rate 自体の文法、および RecipeSystem.md の accepts（型・個数の制約）との関係は GameElementDefinition.md を参照して ください。

本ドキュメントは検討結果であり、確定仕様書ではありません。未決事項は 4 節に整理しています。

1. 重さの伝播は move_to_slot の副作用として扱う

コンテナ自体の重さは、中に入っているアイテムの重さの影響を受けます（バックパックの中身が増えれば、バックパック 自体が重くなる）。これは計算式（derived）ではなく、既存の move_to_slot（唯一の汎用操作）に「アイテムが スロットに出入りするたびに、そのアイテムの実効重量をスロットの持ち主の weight へ加減算する」という不変条件を 1つ持たせるだけで実現します。

これは温度・湿度（ClimateSystem.md）で採用した「毎 tick 加減算する」方式とは異なり、スロット移動という離散 イベントの度に1回だけ加減算する方式です。新しいトリガー体系を追加するのではなく、既存の move_to_slot の 不変条件を1つ増やすだけであり、「値は独立変数への加減算によって駆動される」という数値積分の哲学から外れていません。

入れ子（アイテム→バッグ→バックパック→装備）も、この仕組みが再帰的に働くことで自然にカスケードします。バッグ 自身の weight は、バッグの中身の出入りに応じて常に最新化されているため、バッグをバックパックに入れる瞬間は 「バッグの現在の weight」を1回伝播させるだけで済みます。

2. weight_rate は「身につけると軽い」も「軽減能力の高い箱」も同じ仕組みで説明する

「ほとんどのアイテムは手荷物より身につける方が軽い」という現象と、「台車は中身が増えても重さが増えにくい」と いう現象は、どちらも weight_rate（スロットごとの伝播率）で説明できます。

object_defs:
  character:
    slots:
      equip:
        weight_rate: 0.5    # 身につけると重さが半分に感じる
      inventory:
        weight_rate: 1.0    # 手荷物はそのまま
  cart:
    props:
      weight:
        value: 20
    slots:
      storage:
        weight_rate: 0.1    # 中身の重さの10%しか伝播しない

バックパックを背負うと軽く感じる、という現象も、character.equip と backpack.storage 両方の weight_rate が 効いた結果として説明できます。コンテナごとに専用の「軽減能力」という別概念は作りません。

3. 行動力への影響

キャラクターの合計重量（各スロットからの伝播の総和）を weight のようなプロパティとして持たせ、既存の stages （GameElementDefinition.md 6.4 節）で「軽い/やや重い/かなり重い/動けない」のような段階に分けます。あとは move アクションの条件・効果として、重量段階を通常通り参照するだけです。

actions:
  move:
    conditions:
      - {path: actor.encumbrance_stage, op: neq, value: immobile}

移動時間の延長は各段階の passive の modify、疲労の増加は毎 tick 蓄積する accumulate で表現すればよく、 新しい仕組みは不要です。重さの伝播自体は、どのオブジェクトの slots でも同じように起きます。「行動力に影響する」 のはキャラクター自身の weight を読む stages/actions だけなので、地面に置かれた宝箱の重さが誰の行動力にも 影響しないのは自然な帰結です。

4. 未決事項・今後の検討課題

  • 既に中に入っているアイテムの重さが後から変化した場合（例: 松明の燃料が減って軽くなる）、move_to_slot は 出入りの瞬間にしか発火しないため、入れっぱなしのまま重さが変わってもコンテナ側の weight は自動的に追従 しない。これを許容する仕様上の割り切りとするか、何らかの再伝播の仕組みを別途用意するか
  • size/capacity が単純なスカラー数値で十分か、それとも将来的に形状（テトリス型インベントリ等）を考慮したいか
  • weight_rate が 1.0 を超える（かえって重く感じる）ケースを想定する必要があるか
  • size/capacity/weight_rate の具体的な数値・単位（本書のサンプルはあくまで説明用）
  • 重さの伝播（1 節）と RecipeSystem.md の accepts から自動生成される combinations（材料投入）が、同じ move_to_slot イベントに対してどちらも副作用として発火する場合の実行順序
