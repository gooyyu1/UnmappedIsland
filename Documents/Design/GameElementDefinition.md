# ゲーム内要素の定義方式

## 1. 位置づけ

本ドキュメントは、ゲーム内要素を YAML で記述するための設計を、以下の 3 つの観点からの議論を統合して整理したものです。

- 全体方針（何を、なぜ YAML で表現するか）
- ファイル全体・型定義まわりの具体的な YAML 構造（`WorldCodex` / `object_defs` / `traits` / `props`）
- オブジェクト同士の影響関係を表す効果システム（親子スロット、装備の排他制御、`passive`/`active`）

いずれも粒度の異なる議論の積み重ねであるため、確定した仕様と未決定の論点が混在しています。
「確定事項」は議論済みの決定として記載しますが、「未決事項」は結論が出ていないため、各節および末尾の
12 節に **TODO: 未決定** として選択肢のみを提示します。本ドキュメントは検討結果であり、確定仕様書ではありません。

文中に登場する防具・細菌感染・耐久値などの具体例は、記法を説明するためのサンプルであり、
実際のゲームバランスや病気システムそのものを規定するものではありません。

## 2. 設計思想

### 2.1 YAML定義の全体方針

ゲーム内に登場するほぼすべての概念は **YAML 形式** で定義します。
定義ファイルは複数のファイルに分割して配置できるため、ゲーム本体が提供するデフォルト定義とは別に、
ユーザー（MOD 作成者）がファイルを追加するだけで新しい要素を導入したり、既存の要素を上書き・変更したりすることができます。

管理する概念は、型定義（`ObjectDef`）・実行時インスタンス（`WorldObject`）・全体（`WorldCodex`）の 3 つです。
このうち `WorldObject` 側の詳細フォーマットは未着手であり、本ドキュメントは主に `ObjectDef` 側の構造を扱います。

### 2.2 定義できる概念の種別

**世界レベルの概念**

ゲーム内時刻・日付・季節・天候など、世界全体に影響する概念を定義します。

- 時刻の刻み方（1 tick あたりの経過時間）
- 昼・夜の時刻範囲
- 季節の日付範囲
- 天候の種類と遷移ルール

**ロケーション**

プレイヤーが探索できる場所を定義します。

- 地区（森、海岸、廃村など）
- 建築物・施設
- 洞窟や特殊地形

**プレイヤー・キャラクターパラメータ**

キャラクターが保持すべき数値パラメータを定義します。

- 空腹度、喉の乾き、眠気
- 体温、血液量
- 幸福度、傷みなど

**アイテム**

プレイヤーが入手・使用・製作できる物を定義します。

- 設備・道具
- 資材・食料
- 薬品・素材など

**アイテム属性**

アイテムが持つ性質（燃焼性、腐敗速度、重量など）を定義します。属性はアイテムに付与する形で参照され、アクションの結果に影響します。

**アイテムアクション**

アイテムに対して実行できる操作（食べる、使う、加工するなど）を定義します。アクションは実行可否の条件と、実行結果（`active`）を
1つの定義としてまとめて持ちます（詳細は 8 節、`ActionSystem.md`）。

### 2.3 統一的な実行エンジン

可能な限り、すべての概念は共通の枠組みで表現され、単一の汎用エンジンによって実行されます。
「特定の概念専用のロジック」を増やすことを避け、拡張性と MOD 互換性を確保します。

**tick 駆動による変化**

ゲーム内のあらゆる「時間経過にともなう変化」は tick を基本単位とする同一メカニズムで実現します。

| 現象 | 仕組み |
|------|--------|
| ゲーム内時刻の進行 | 1 tick ごとに「時刻」パラメータを加算 |
| 空腹度の増加 | 1 tick ごとに「空腹度」パラメータを加算 |
| 傷の出血による血液量の減少 | 1 tick ごとに「血液量」パラメータを減算 |
| 部屋の快適度によるキャラクターの気分の変動 | 1 tick ごとに「幸福度」パラメータを加減算 |

**範囲による状態の決定**

パラメータの値域から状態を決定する仕組みも共通化します。

| 現象 | 仕組み |
|------|--------|
| 時刻の範囲で昼夜を決定 | 時刻パラメータの値域で「昼」「夜」を割り当て |
| 日付の範囲で季節を決定 | 日付パラメータの値域で「春」「夏」「秋」「冬」を割り当て |
| 空腹度の範囲で飢えや満腹の状態を決定 | 空腹度パラメータの値域で「満腹」「空腹」「飢え」を割り当て |

このように、単純なパラメータとその値域の定義だけで、世界の挙動もキャラクターの状態も同一の手法で記述できます。
この「値域による状態決定」を型定義レベルで一般化したものが、6.4 節で述べる `stages`（段階）です。

### 2.4 設計上の方針

- 「ハードコードしない」：コードに定数や挙動を埋め込まず、YAML 定義で制御できるようにする
- 「汎用エンジンに任せる」：特定概念向けの例外処理をエンジン側に追加しない
- 「ファイル追加だけで拡張できる」：MOD 作成者がコードに触れることなく新要素を追加できる

## 3. ファイル全体構造とロード

### 3.1 WorldCodexとルート構造

ファイル全体が 1 つの `WorldCodex` を表します。**専用のルートキーは置きません**（ファイル自体が `WorldCodex` です）。

### 3.2 命名規則

- 構造キーワード（`props`、`traits`、`actions` など）・型名・プロパティ名は、**すべて snake_case** で統一します。非プログラマも編集するため、camelCase との混在を避けます。
- id（型名・プロパティ名・トレイト名・アクション名・スロット名に共通）の命名規則は次の正規表現に従います。

  ```
  ^[a-z][a-z0-9_]*$
  ```

### 3.3 パーサ要件（重複キーの扱い）

YAML 仕様上マッピングキーは一意であるべきですが、多くのパーサ実装（素の PyYAML 等）は重複キーをエラーにせず後勝ちで上書きします。
これは MOD 由来の記述ミスなどを静かに握りつぶす危険があるため、以下の二重の安全策を仕様として明記します。

- ローダーは重複キー検出でエラーを出す**厳格モード**を使用する
- 加えて、ロード後に別途**バリデーションステップ**を設ける

### 3.4 YAML定義の分割とロード

YAML ファイルは所定のディレクトリ以下に置かれたすべてのファイルが自動的に読み込まれます。
ゲーム本体の定義は `data/` などの標準ディレクトリに、MOD の定義は別ディレクトリに配置することで、双方を混在させずに管理できます。

同じ ID を持つ定義が複数のファイルに存在する場合のマージ・上書き規則は別途仕様書で定義します。

## 4. object_defs（型定義）

型定義は `object_defs` の下に、識別子を**キーとして**表現します（値ではありません）。

```yaml
object_defs:
  stone:
    props:
      weight:
        value: 10
```

## 5. traits（mixin）

記述量削減のため、複数の `object_def` で共有するプロパティ群を `traits` として定義できます。

- **多重継承は禁止**: trait が他の trait を参照することはできません（1 階層のみ）。
- **プロパティ衝突はエラー**: 複数の trait が同名プロパティを宣言していた場合、暗黙の優先順位で解決せずエラーとします（安全側に倒す）。
- **値の扱い**:
  - trait 側で `value` を持たないプロパティ = object_def 側で値を与えることが必須
  - trait 側で `value` を持つプロパティ = デフォルト値。object_def 側で省略可、上書きも可
  - object_def 側で一部属性（例: `value`）だけを上書きした場合、残りの属性（`unit` など）は trait 側の値を引き継ぐ

```yaml
traits:
  perishable:
    props:
      shelf_life:      # valueなし→継承先で必須
        unit: days
      decay_rate:
        value: {min: 1, max: 3}   # valueあり→デフォルト値(範囲値は6.2節参照)

object_defs:
  apple:
    traits: [perishable]
    props:
      shelf_life:
        value: 7
```

アクション・効果も trait 経由でまとめて配布できます（7〜8 節参照）。例えば `eatable` trait が `eat` アクションと関連 props をセットで提供する、といった使い方を想定します。

## 6. props（プロパティ）

### 6.1 固定値

最も単純な形は、`value` に固定値を 1 つ持つだけの形です（4 節の `stone.props.weight` を参照）。

### 6.2 範囲値（ランダム再ロール）

固定値の代わりに `{min, max}` の範囲を `value` に持たせ、**毎 tick ごとに範囲内でランダムに値を再ロール**できる仕組みを持ちます。
腐敗速度・天候の持続時間など、複数の用途で共有する汎用プリミティブとして設計します。

```yaml
props:
  decay_rate:
    value: {min: 1, max: 3}   # 毎tick、この範囲でロールした量を減算する
```

> 持続時間を `{min, max}` の範囲値として表現すること自体は本節の仕組みを流用できますが、
> 「どの天候に遷移するか」の選択ロジック（重み付き抽選など）は別途検討が必要です。

### 6.3 overflow（周回・上限のあるプロパティ）

時刻のように上限に達したら折り返す（あるいは繰り上げる）プロパティのため、`range` と `on_overflow` を持たせます。
`value` の `{min, max}`（6.2 節、毎 tick 再ロールする範囲）とは異なる仕組みである点に注意してください。
`range` は値が取りうる上下限、`on_overflow` は上限到達時の挙動を表します。

```yaml
props:
  minute_of_day:
    value: 0
    range: {min: 0, max: 1439}
    on_overflow: {mode: wrap, carry_to: day}   # 1440でdayに+1して0に戻る
  sequence:
    value: 0
    on_overflow: {mode: none}                   # 上限なし
```

> - 案A: `day` は無制限に加算され続ける（`on_overflow: {mode: none}`）
> - 案B: `day` は年単位（例: 360 日）で `year` に繰り上げて wrap する（`on_overflow: {mode: wrap, carry_to: year}`）

### 6.4 stages（段階）

数値パラメータ（プロパティ）は `stages` を持つことができ、現在値がどの段階にあるかによって、有効になる `passive`（8 節）が切り替わります。

- 段階の区間定義は **`min`（下限）のみを指定します**。上限は明示せず、次の段階の `min` によって自動的に決まります（半開区間、`[min, 次の min より手前]`）。この方式により、区間の隙間や重複が構造的に発生しません。
- 最下段の段階は `min: null`（または省略）とし、それより下の残り全ての値を拾います。
- 現在値に基づいて常に一意に段階を決定します。ヒステリシス（上昇時・下降時で閾値をずらす仕組み）は採用しません。プレイヤーが常に現在の状態から結果を逆算できるようにし、状況分析をしやすくするための設計意図です。
- ステージの `passive` はレベルトリガー（そのステージにいる間ずっと有効）を基本とします。ステージ切替の瞬間だけ発火する汎用的な仕組み（edge-triggered、例えば `on_enter` 的なもの）は導入していません。「値が0になった瞬間に何かを起こす」という、実際に必要とされる唯一のケースは、汎用的なステージ出入りイベントではなく専用の `on_zero`（6.6 節）で表現します。

```yaml
properties:
  durability:
    value: 100
    stages:
      - name: intact
        min: 1
      - name: broken
        min: null
```

具体的な `passive` と組み合わせた例は 11.2 節を参照してください。

### 6.6 on_zero（0に達した瞬間のイベント）

`stages`（6.4 節）はレベルトリガーであるため、「ステージに入った瞬間だけ」発火する仕組み（edge-triggered）を
表現できません。実際に必要になる場面を洗い出すと、その大半は「値が正から0以下に落ちた瞬間に何かを起こす」
（耐久値が尽きたら破棄する、季節の残り tick が尽きたら次の季節へ切り替える、等）という、ゼロという1点に
限定された単純なケースでした。汎用的な「ステージへの出入り」イベント（`on_enter`/`on_exit`）を用意する
代わりに、この限定的だが実際に必要なケースだけを扱う `on_zero` を導入します。命名は `on_overflow`（6.3 節、
上限側の折り返し）と対になる、下限側の1点イベントという位置づけです。

- `on_zero` はプロパティが**正の値から0以下へ跨いだ、そのtickだけ**発火します。既に0以下の状態が続いている
  間は再発火しません。跨いだかどうかは前 tick との符号比較のみで判定でき、任意個のステージ境界を跨いだかを
  追跡する必要がある一般的な edge-triggered の仕組みより実装が単純です。
- `on_zero` の中身は `active`（8.2〜8.3 節）と同じ対象キーの辞書を直接持ちます。`conditions` のような兄弟
  フィールドが今のところ不要なため、`passive` がオブジェクトレベルで対象キー辞書を直接持つのと同様、
  余分なラップを挟みません。

```yaml
props:
  durability:
    value: 100
    range: {min: 0, max: 100}
    accumulate:
      self:
        durability: -1
    on_zero:
      self:
        lifecycle:
          destroy: true
```

`on_zero` の導入により、`lifecycle` は `active` 専用に戻せます（8.3 節参照）。耐久値が0になったら破棄する、
というこの用途こそが、これまで `lifecycle` をステージの `passive` に書かせてしまっていた唯一の実例だった
ためです。一般的な `on_enter`/`on_exit` を用意するかどうかは、まだ議論の余地があります（12 節参照）。今の
ところ実例が `on_zero` で説明できるケースしかないため、必要性が生じるまでは導入しません。

## 7. オブジェクトの親子関係とスロット

### 7.1 親子関係とスロットの基本構造

- オブジェクト間の所属関係はツリー構造で表現します。1 つの子オブジェクトは必ず 1 つの親に属し、複数の親を持つ関係は採用しません。
- 親オブジェクトは複数の「スロット」（配列）を持ちます。例: `inventory`（所持品）、`equip`（装備）、`fixtures`（部屋の備え付け設備）、`injuries`（怪我）。
- 子オブジェクトは `held_in` のような逆引き用のキャッシュフィールドを持ちますが、正の情報源はあくまで親側のスロット配列です。
- スロット間の移動は `move_to_slot` という単一の汎用操作に集約します。親側スロット配列と子側の逆引きキャッシュの整合性は、この操作の中でのみ保証します。個別の YAML 定義側が両者を直接書き換えることは想定しません。
- スロットに「種類タグ（kind）」は設けません。`equip` はそれ自体が唯一の装備用スロットであり、スロット名を直接参照します。スロットの意味をタグで抽象化するよりも、名前をそのまま参照する方が単純で、現時点で複数の装備スロットを区別する必要も生じていないためです。

```yaml
object_defs:
  character:
    props:
      hp:
        value: 100
    slots:
      equip: {}
      inventory: {}
      injuries: {}
```

### 7.2 装備の排他制御（layer / covers）

- `equip` スロットには複数のアイテムを同時に格納できます。排他制御はスロットの数ではなく、アイテムが持つ属性によって行います。
- アイテムは `covers`（覆う部位のタグの配列。例: `[torso]`、`[legs, waist]`）と `layer`（重ね着の階層。例: `base`、`outer`）を持ちます。
- 競合判定は、同じ `(covers要素, layer)` の組が、既に `equip` スロットに入っている他アイテムと重複する場合に装備不可とします。
  - 例: シャツ（`torso, base`）とジャケット（`torso, outer`）は共存可能です（layer が異なるため）。
  - 例: シャツ（`torso, base`）とつなぎ（`legs/waist/torso, base`）は共存不可です（`torso, base` が競合するため）。
- 競合解決は「ブロック型」を採用します。競合するアイテムがある場合、プレイヤーが先に既存の装備を外す必要があります（自動退避は採用しません）。装備の入れ替えを明示的な操作として要求することで、プレイヤーが装備状態を把握しないまま暗黙に変更されることを避けるためです。

## 8. アクションと効果（actions / passive / active）

### 8.1 actions（メニュー型の宣言的操作）

- アクション（`eat`、`move` など）は、条件（`conditions`）と実行結果（`active`）を**1つの定義としてまとめて持ちます**。かつては
  「条件だけを持つ受動的な宣言」と「`on: <アクション名>` で別途発火する効果」を分離していましたが、1つの操作の挙動が複数箇所に
  分散して読みにくくなるため、この分離は廃止しました（詳細な経緯・カード間の相互作用（ドラッグ操作）は `ActionSystem.md` を参照）。
- アクションは `object_defs` および `traits` の中に配置します（トップレベル独立キーにはしません）。これにより、traits 経由でアクションもまとめて配布できます。
- `showMenu` でメニューへの表示方法を制御します（現時点では `always` のみ。将来的にボタン以外の起動方法が増えた場合に備え、列挙可能な形にしています。詳細は `ActionSystem.md`）。
- `active` は、このアクションが実行された瞬間に 1 回だけ適用されます（8.3 節参照）。

```yaml
traits:
  eatable:
    actions:
      eat:
        showMenu: always
        conditions:
          - {path: actor.satiety, op: lt, value: max}
        active:
          actor:
            add:
              satiety: 10
          self:
            lifecycle:
              destroy: true
```

明るさで移動可否を制御する例です。

```yaml
object_defs:
  world:
    singleton: true
    props:
      light_stage:
        value: bright   # 6.5節の議論次第でderivedになる可能性あり(未確定)

  character:
    props:
      hp:
        value: 100
    actions:
      move:
        showMenu: always
        conditions:
          - {path: world.light_stage, op: in, value: [bright, dim]}
```

条件式は `{path, op, value}` の形を取ります。`path` は参照ルートから始まるドット区切りのパスです。
現時点で定義されている参照ルートは `self`（このアクションを宣言したオブジェクト自身）・`parent`（その親）・
`actor`（このアクションを実行しているプレイヤーキャラクター）・`world`（環境シングルトン、9 節参照）の 4 つです。
`self`/`parent`/`actor` は 8.2 節の対象キーと同じ語彙・同じ意味で使います。

**`actor` はすべてのアクションに暗黙的に存在し、常にプレイヤーキャラクターを指します。** `eat` の例のように、
アクションを持つオブジェクト（食べ物）自身は消費される側（`self`）であり、実際に効果を受け取るのは食べ物を保持している
容器（`parent`）ではなく、行動主体である `actor` です。`parent` はあくまで木構造上の直接の格納先を指すものであり、
`actor` とは独立した参照です（例えば地面に置かれた食べ物を直接食べる場合、`parent` はキャラクターではなく地面のロケーションになりますが、
`actor` は変わらずプレイヤーキャラクターを指します）。

カードを別のカードにドラッグする操作（例:「斧で木を切る」）は、`actions` ではなく `combinations` という別の仕組みで表現します。
一方、攻撃のように、行動主体（`actor`）以外に登場する参加者が「メニューを開いたカードそのもの」（例: 攻撃対象の敵カード）
だけで済む場合は、ドラッグ操作を使わず `actions`（メニュー型）で表現します（詳細は `ActionSystem.md`）。

### 8.2 passiveの基本構造

`passive` は次の 3 つのレベルで定義でき、いずれも同一の記法・実行原理を用います。

1. **オブジェクトレベル**（例: 防具そのものが持つ複数の恩恵の束）
2. **プロパティレベル**（例: アイテムの重量プロパティが持ち主の負荷に寄与する）
3. **プロパティのステージレベル**（例: 耐久値の残量帯によって防御効果が変わる。6.4 節参照）

`passive` は、**効果の対象（`self`/`parent`/`child`/`actor`）を識別子とするキー**として表現する辞書型です。
現時点で定義するのは `self`（自分自身）・`parent`（親）・`child`（子）・`actor`（このアクションを実行しているプレイヤーキャラクター、8.1 節参照）の 4 つです。
`ancestor`（先祖）・`sibling`（兄弟）・`descendant`（子孫）は、必要性が生じた時点で改めて検討します。

```yaml
passive:
  parent:
    when: equip
    modify:
      defense: 5
      speed: 3
      accuracy: 2
```

- `when: <スロット名>` … そのスロットに入っている間、継続的に有効（レベルトリガー）
- `when` を省略した場合は「常時（無条件）」を意味します。

`when` は各対象（`self`/`parent` など）のキーの中に書きます。1 つの `passive` の中で、対象ごとに異なる `when` を
持たせることもできます（例: `parent` は `when: equip` の間だけ、`self` は常時、など）。トリガーを `passive` 全体の
兄弟キーに引き上げる案も検討しましたが、同じオブジェクトが複数の異なるトリガーを将来持つ可能性を残すため、
対象ごとに `when` を持たせる現在の形を採用しています。

`active`（8.1 節・8.3 節）も `self`/`parent`/`child`/`actor` という同じ対象キーの辞書型を使いますが、`when` は持ちません。
`active` は持続する条件を表す必要がなく、アクション・組み合わせ・確率分岐の結果が確定した瞬間に無条件で1回だけ
適用されるためです。

特定のアクションが実行された瞬間にだけ発火する効果は、`on: <アクション名>` のような独立したトリガーとしては表現しません。
アクション自身の `active`（8.1 節）、またはカード間の組み合わせを表す `combinations` の `active`（`ActionSystem.md`）として、
アクションの定義そのものに直接書きます。

### 8.3 効果の種別（modify / accumulate / add / lifecycle）

効果は「持続する影響（`passive` に書く）」と「一時的な命令（`active` に書く）」という性質で大きく2つに分かれ、
値を操作する動詞・存在を操作する動詞のいずれも、どちらか一方の性質にのみ属します。

**持続する影響（`modify`/`accumulate`、`passive` にのみ書く）**: `self`/`parent`/`child` の関係とゲート（常時／
`when: <スロット名>`／プロパティの stage）に紐づいて登録され、その関係が続く限り評価され続けます。違いは値の可逆性だけです。

- **`modify`**: 条件が真である間だけ計算式に寄与する、可逆的な修飾子です。条件が偽になった瞬間に寄与は自動的に消えます。値そのものを書き換えるのではなく、実効値を都度導出するための入力として扱います。
  - 同一プロパティに対する複数の `modify` は単純加算で合成します。
  - 合成結果の上限・下限（min/max）はプロパティ（スタット）側の定義に持たせます。
- **`accumulate`**: 条件が真である tick 毎に、対象プロパティの実体値へ直接、不可逆に加減算します（例: 出血中の血液量減少、耐久値の毎 tick 減少）。

**一時的な命令（`add`/`lifecycle`、`active`/`on_zero` にのみ書く）**: `actions`/`combinations`（8.1 節、`ActionSystem.md`）や
`pick`（`PickSystem.md`）の `active`、または `on_zero`（6.6 節）として書かれ、それぞれの結果が確定した瞬間・プロパティが
0に達した瞬間に一度だけ実行されます。持続する条件を表す `when`/ゲートは最初から不要で、`modify`/`accumulate` のような
登録の仕組みにも乗りません。

- **`add`**: 対象プロパティの実体値へ、その場で一度だけ不可逆に加減算します。`when`/常時の継続的な加算は `accumulate` が
  担うため、`add` は一時的な命令専用に意味を絞り込んでいます（後述）。
- **`lifecycle`**: プロパティの値ではなく、オブジェクトそのものの存在を操作します（例: `destroy` でオブジェクトを消滅させる、`spawn` で新規オブジェクトを生成する）。値操作である `modify`/`accumulate`/`add` に対し、`lifecycle` は存在操作を担う独立したカテゴリとして区別します。
  - `lifecycle` は複数の動詞を同時に持てる辞書形式です（例: `{spawn: {...}, destroy: true}`）。同じ対象に対して「新しいオブジェクトを生成しつつ、自分自身は消滅させる」という組み合わせを1つのエントリで表現できます。
  - `lifecycle` は一時的な命令専用であり、`passive` には書けません。「値が0になったら破棄する」（11.2 節の耐久値サンプル等）という、これまで唯一 `lifecycle` を `passive`（ステージ）に書かせていた用途は、汎用的なステージ出入りイベントではなく専用の `on_zero`（6.6 節）で表現します。

以前は `add` という1つのキーワードで「アクション実行に伴う効果なら一回、`when`/常時なら毎 tick」という2つの適用頻度を
書く場所だけで使い分ける設計でしたが、実装を検討した結果、この2つは仕組みが大きく異なることが分かりました。`when`/常時の
継続的な加算は、`modify` と全く同じ「関係とゲートに基づく登録」の仕組みを流用でき（違いは可逆か不可逆かだけ）、一方
アクション実行時の加算は登録もゲート判定も不要な、その場限りの直接呼び出しで済みます。この実装上の違いをキーワード自体が
表すよう、継続的な加算には初期検討にあった `accumulate` を割り当て、`add` はアクション/combination/pick/on_zero専用の
一時的な命令として意味を絞り込みました。同じ理由から、この2つを書く場所（コンテナ）自体も `effects` という1つの
キーワードで共用するのをやめ、`passive`（`modify`/`accumulate`）と `active`（`add`/`lifecycle`）に分けています。この2つが
同じ場所に混在することは構造上ないため（`object_defs`/`props`/`stages` 直下は常に `passive`、`actions`/`combinations`/
`pick`/`on_zero` は常に `active`）、コンテナ名を分けても表現力は失われません。

`modify`/`accumulate`/`add`/`lifecycle` はいずれも、対象（`self`/`parent` など）ひとつに対して「何が起きるか」を確定的に記述するものです。
複数の結果から重み付き確率で1つを選ぶ必要がある場合は、`active` を書く代わりに `pick` を使います。`pick` は
`active` と同じ場所に書く**代替の**キーであり、各候補が自分自身の `active`（対象ごとの辞書）を丸ごと持てます。
これにより「確率で分岐した結果、候補ごとに異なる対象へ効果が及ぶ」ケースも表現できます。詳細は `PickSystem.md` を参照してください。
なお `pick` はアクション/combinationの解決と同じ「一時的な命令」の瞬間に一度だけ発生するものなので、候補の `active` に
書く意味があるのは一時的な命令（`add`/`lifecycle`）だけです。`modify`/`accumulate` は関係とゲートに基づいて登録され、
その関係が続く限り評価され続けることに意味がある仕組みのため、1回限りで消える `pick` の候補に書いても、条件が
再評価されない値になってしまい、`add` と区別がつかなくなります。この点も含め、`pick` の位置づけ自体は今後の
検討課題です（12節参照）。

### 8.4 カード間の相互作用（menu / combinations）

プレイヤーがカードに対して行う操作は、大きく2種類に分かれます。

- **メニュー型**: 1枚のカードだけで完結する操作。8.1 節の `actions` として、条件と効果をまとめて宣言します。すべてのアクションには
  `actor`（プレイヤーキャラクター）が暗黙的に参加できるため、`eat`（食べ物 self ＋ actor）だけでなく、`attack`（攻撃対象の敵 self ＋ actor）
  のように、一見「2つの参加者が要る」ように見える操作も、もう一方が常に `actor` であればメニュー型で表現できます。
- **ドラッグ型**: あるカードを、`actor` ではない別の特定のカードにドラッグ＆ドロップして行う操作（「斧で木を切る」など、
  道具や素材そのものを指定する必要がある場合）。ドロップされた側（受け側）の `combinations` として宣言します。

ドラッグ型の詳細な記法（`combinations` の構造、ドラッグされてきたカードを指す `dragged` という参照、MOD 拡張性、
対称的な組み合わせの扱いなど）は `ActionSystem.md` に切り出しています。

以前は、アクションが成立する条件と、成立した結果として起きる効果を、`on: <アクション名>` という独立したトリガーを介して
別々に記述する案を検討していました。これは「アクションを宣言していない第三者も、そのアクション名を共有トリガーとして
listen できる」という拡張性を狙ったものでしたが、1つの操作の挙動が複数のオブジェクトに分散して記述されることになり、
挙動を把握しにくくなる（Wiki 等での文書化がしづらい）という欠点が大きいと判断し、採用しませんでした。
この判断の詳細な経緯は `ActionSystem.md` に記載しています。

## 9. singletonオブジェクト（world）

ゲーム動作に必須ないくつかの情報（日時、天候、明るさ等）を持つ、唯一のインスタンスを想定しています。
「インスタンスが 1 つだけ存在すべき」という制約は、専用のルートキーを作るのではなく、`object_defs` のエントリに小さなフラグ（`singleton: true`）を追加することで表現します。

```yaml
object_defs:
  world:
    singleton: true
    props:
      sequence:
        value: 0
        on_overflow: {mode: none}
      day:
        value: 1
        on_overflow: {mode: none}   # TODO: 未決定。年単位でのwrapについては6.3節を参照
      minute_of_day:
        value: 0
        range: {min: 0, max: 1439}
        on_overflow: {mode: wrap, carry_to: day}
      weather:
        value: clear
```

`sequence`・`day`・`minute_of_day` は 1 つの複合値にまとめず、独立した props として持ちます。
6.3 節の overflow 機構は prop 単位（`range` / `on_overflow`）で定義される仕組みであり、複合値の内側の要素それぞれに
overflow 設定を紐づける手段は存在しないためです。

日時・天候はオブジェクトから直接参照されるのではなく、**環境がオブジェクトに影響を与える**という位置づけです（例: 明るさによって行動可否が変わる）。
直接のプロパティ参照経路（`world.xxx` のようなパス表記）は 8.1 節の action 条件式で使う想定です。

## 10. terrain / 島生成

地形は `object_defs` の一種として表現しますが、島の自動生成アルゴリズムが必要とする属性群があります。
これは新しい構造を作らず、**trait として必須 props を宣言する**ことで表現します。

```yaml
traits:
  terrain_candidate:
    props:
      generation_weight: {}   # valueなし→object_def側で必須

object_defs:
  forest:
    traits: [terrain_candidate]
    props:
      generation_weight:
        value: 5
```

「value なしプロパティ」は、5 節の trait の値継承ルールに従い、空のマッピング（`{}`）として表現する形に統一しています。

上記はあくまで「trait で必須 props を宣言する」という仕組みの一般例であり、実際に地形（`LocationType`）を
どう選定するかの具体的なアルゴリズムは `TerrainGeneration.md` で軸ベースの最近傍マッチングとして別途定義しています。

## 11. サンプル集

以下は、ここまでの記法を組み合わせた説明用のサンプルです。実際のゲームバランスを規定するものではありません。

### 11.1 防具（オブジェクトレベルの効果、複数の恩恵の束）

```yaml
object_defs:
  armor_leather:
    covers: [torso]
    layer: base
    passive:
      parent:
        when: equip
        modify:
          defense: 5
          speed: 3
          accuracy: 2
```

### 11.2 耐久値（毎 tick 減少、on_zero による消滅、ステージによる状態表現）

```yaml
properties:
  durability:
    value: 100
    range: {min: 0, max: 100}
    passive:
      self:
        accumulate:
          durability: -1
    on_zero:
      self:
        lifecycle:
          destroy: true
    stages:
      - name: intact
        min: 1
      - name: broken
        min: null
```

耐久値が尽きたときの破棄は、ステージではなく `on_zero`（6.6 節）で表現します。`stages` は「今 `intact`/`broken`
のどちらか」という状態表現（UI 表示や、将来 `broken` の間だけ防御力を下げるような `passive` の `modify` を
追加する際の土台）として引き続き独立に持たせます。

### 11.3 細菌感染（オブジェクト化された「条件」、段階に応じた多段の影響）

感染度自体をキャラクターの生パラメータにはせず、`conditions` のようなスロットに格納される独立したオブジェクトとして扱います。
怪我・疲労などの「条件」は複数同時に存在しうるため、条件ごとに独立したオブジェクトとして表現する方が、キャラクター側のパラメータを増やし続けるより見通しが良いためです。

感染オブジェクトの `progress` プロパティが `stages` を持ち、段階ごとに体温・水分・意識などの他プロパティへ影響します。

```yaml
condition: infection
properties:
  progress:
    stages:
      - name: none
        min: 0
      - name: mild
        min: 20
        passive:
          parent:
            accumulate:
              temperature: 0.1
      - name: feverish
        min: 50
        passive:
          parent:
            accumulate:
              temperature: 0.2
              hydration: -1
```
