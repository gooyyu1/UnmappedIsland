# レシピシステム設計

## 概要

本ドキュメントは、2つ以上のアイテムを組み合わせて新しいアイテムを作成する仕組み（道具の生成・建物の建築の両方に使う）を
検討した結果をまとめたものです。

呼称は「ブループリント」と「レシピ」を検討し、**レシピ**を採用しました。`blueprint` は大規模建造物・乗り物など工学的な
「設計図」を指す語感が強く、道具のような単純なものの生成にまで使うとやや大仰に響きます。`recipe` はクラフト系ゲームで
「N個の材料→1つの成果物」という組み合わせ全般を指す、より一般化された語彙であり、道具の生成と建物の建築を1つの
統一されたメカニズムとして扱いたいという狙いに合致します（`ActionSystem.md` の `combinations` 命名検討時、この用途のために
`recipe` という語を温存する判断をしています）。

本ドキュメントは検討結果であり、確定仕様書ではありません。未決事項は 9 節に整理しています。

## 1. 基本構造

レシピは**完成品（成果物）の `object_defs` に埋め込みます**。`actions`/`combinations` が「このオブジェクトは何ができるか」
「このオブジェクトは何と組み合わさるとどうなるか」を、そのオブジェクト自身の定義の中に持つのと同じ理由です。レシピも
本質的には「このオブジェクトはどうやって作られるか」という、成果物であるオブジェクト自身に紐づく情報だからです。

1つのアイテムに複数の作り方（材料違いのレシピ）がありうるため、`combinations` と同様、名前をキーとする辞書型にします。

```yaml
object_defs:
  axe:
    recipes:
      basic:
        icon: axe_wip.png
        steps:
          - requires:
              - { object: wood, quantity: 2, consume: true }
              - { object: stone_knife, consume: false }
            duration: 30
          - requires:
              - { object: rope, quantity: 1, consume: true }
            duration: 10
```

- レシピは 1 つ以上の**工程（`steps`）**からなります。
- 各工程は 1 つ以上の**素材または道具**を要求します（`requires`）。素材（`consume: true`）は消費され、道具（`consume: false`）は
  消耗しますが消費はされません。
- 各工程には所要**時間（`duration`）**が定義されます。
- 最後の工程まで完了すると、目的のアイテムが生成されます。

## 2. 製作中オブジェクトは、完成品とは別オブジェクト

「未完成の斧」を、斧自身の `stages`（例: `progress` プロパティの段階）として表現する案も検討しましたが、採用しませんでした。

- 斧に複数のレシピがある場合、「今どのレシピを実行中か」「何工程目か」という情報を、斧自身の型定義に混ぜ込むことになります。
- 「0% 完成の斧」は実質的には材料の集まりに近く、斧という同一性を最初から持たせるのは不自然です。

代わりに、**製作中の状態専用の、別オブジェクト**として扱います。これは `infection`（`GameElementDefinition.md` 11.3 節）や
季節（`ClimateSystem.md`）で採用した「独立した条件・プロセスオブジェクト」パターンと同型です。製作中オブジェクトは
`progress` プロパティに `stages`（工程に対応する半開区間）を持ち、最終工程の効果として
`lifecycle: {spawn: {object: axe}, destroy: true}`（自分を破棄し、完成品を生成する）を発火させます。

### 2.1 製作中オブジェクトの型は自動生成する

製作中オブジェクトの型を、レシピごとに人間が手で書く（`axe_in_progress` のような `object_defs` を個別に用意する）案と、
全レシピ共通の汎用型を1つ用意する案の両方を検討しましたが、どちらも採用しませんでした。

- **手書きの個別型**: 全ての製作中オブジェクトが同じ挙動をすべきところ、著者ごとに実装がばらつく余地が生まれます。
- **単一の汎用型**: 3 つの問題があり、いずれも「インスタンス生成時にプロパティを個別に注入する仕組みがない」という、
  同じ根本原因に行き着きます。
  - 特定のプロパティ（目指す完成品は何か、等）が必ず必要という、エンジン側の暗黙の契約が生まれる
  - 完成品ごとに異なるアイコンなど、インスタンスごとの見た目を変える手段がない
  - レシピに応じたパラメータを持たせてインスタンスを生成する仕組みが既存にない

代わりに、**レシピの内容（工程数・各工程の要求・時間）から、決まった変換規則で製作中オブジェクトの型を自動的に導出**します。
人間も MOD 作成者も、この型を直接書くことはありません。アイコンなど、完成品ごとに変えたい情報は、レシピ自身のデータの
一部として著者が指定し（1 節の `icon` 参照）、自動生成時にその情報を型へ引き継がせます。これにより、インスタンス生成時の
パラメータ注入という、まだ手つかずの `WorldObject` 側の設計（`GameElementDefinition.md` 2.1 節）に踏み込まずに済みます。

この「決まった変換規則で型を導出する」という考え方自体、`GameElementDefinition.md` が掲げる「汎用エンジンに任せる」
（概念ごとの特別なロジックを増やさない）という方針の、また別の適用例です。特定のアイテム向けのハードコードではなく、
あらゆるレシピに対して同じ1つの生成規則を適用する、という一般化です。

## 3. 進捗: 既存の stages を流用

工程の進み具合は、既存の `stages`（`GameElementDefinition.md` 6.4 節）にそのまま乗ります。数値プロパティ（`progress`）と
半開区間、という組み合わせが「今どの工程か」を表すのに無理なく合致するためです。

## 4. 材料・道具の要求: 新設のスキーマ

一方、工程ごとの材料・道具の要求（`requires`）は、既存の `conditions`/`effects` には当てはまりませんでした。

- `conditions`（`{path, op, value}` の AND リスト）には、**工程ごとに異なる要求内容**を表現する仕組みがありません。
- `effects`（`modify`/`accumulate`/`add`/`lifecycle`）にも、「在庫を確認し、指定数量だけ消費し、足りなければ何もしない」という
  条件チェック＋消費の複合動作を表す語彙がありません。

これを無理に既存の語彙へ押し込めるより、`steps`/`requires` という**専用のスキーマ**を新設する方が良いと判断しました。
これは既存の設計方針（ハードコードしない）への違反ではありません。`on_overflow`、装備の `covers`/`layer` マッチング、
地形の `axis_preferences` なども、それぞれ専用のスキーマと専用の解釈ロジックを持っています。「ハードコードしない」は
「特定のアイテムだけの特別扱いをコードに書かない」ことを指しており、あらゆるレシピに対して同じ解釈ロジックを適用する
専用スキーマを新設すること自体は、この方針に反しません。

## 5. レシピは特殊な slot（箱）として表現する

ゲーム上、レシピは「箱」として扱います。必要な素材・道具を箱に入れてからボタンを押すと、素材が消費されて進捗が進みます。

この「箱」は、既存の `slots`（`GameElementDefinition.md` 7.1 節）の未決事項にちょうど接続します。7.1 節には以前から
「slot に容量・サイズ制限を持たせるか」「slot に格納可能なオブジェクトの型制約の指定方法」という 2 つの未決事項が
残っており、レシピの「箱」はこの両方が同時に必要になる、初めての具体的なケースです。そこで、レシピ専用の新しい
コンテナ概念を作るのではなく、**`slots` に任意の `accepts` フィールドを追加する形で拡張**します。

```yaml
slots:
  inventory: {}          # 制約なし(既存通り)
  equip: {}              # 制約なし(既存通り)
  materials:
    accepts:
      - { object: wood, max: 2, consume: true }
      - { object: stone_knife, max: 1, consume: false }
```

- `accepts` を省略すれば、これまで通り何でも入る無制限のスロットです（`inventory`/`equip`/`fixtures` は変更なし）。
- `accepts` を書けば、型ごとの上限数を持つスロットになります。レシピの「箱」はこの形の `slots` そのものです。

普通の箱にも緩やかな制約（家は入らない）はあるはずなので、`accepts` を「特定の型・個数まで」だけでなく「特定の
trait・サイズまで」のような緩い条件にも使える汎用の仕組みにしておけば、普通の箱とレシピの箱を**同じ機構の厳しさの違い**
として説明できます。完全に別物ではないが同じでもない、という位置づけです。

## 6. 投入操作: accepts から自動生成される combinations

箱へのアイテム投入は、製作中オブジェクト（2.1 節、自動生成される型）に対して、`accepts` の各エントリから**自動的に
`combinations` を生成**する形で実現します。

- `wood` をドラッグ → `accepts` の `wood` エントリにマッチ → 現在の投入数が `max` 未満なら成立 → `materials` スロットへ移動
- `stone_knife` も同様

これは新しい実行モデルではなく、既存の `combinations`（`ActionSystem.md`。受け側が `with` で相手を判定し、条件・効果を持つ）を
そのまま流用し、人間が書く代わりにエンジンが `accepts` から生成するだけです。製作中オブジェクトの型自体が自動生成である、
という 2.1 節の結論と一貫しています。

## 7. 消費・進捗判定: 通常の actions、ただし中身の検査・消費ロジックはプログラム側

「箱に必要なものが揃っているか」を確認し、ボタン（`actions.work` 相当）を押すと素材を消費して進捗を進める、という部分は、
通常の `actions` の `conditions`/`effects` として表現します。

ただし、**「スロットの中身をまとめて検査し、条件を満たせば該当分だけ消費する」という具体的な処理は、レシピでしか
使わない概念であるため、YAML の汎用語彙としては表現せず、プログラム側の専用ロジックとして実装します。** `conditions` の
演算子（`op`）を増やしたり、`effects` に新しい種別を追加したりはしません。これは、レシピの `steps`/`requires` という
データ自体は宣言的・MOD 拡張可能な YAML のまま保ちつつ、それを解釈する具体的な処理コードは、地形生成の軸マッチングや
装備の `covers`/`layer` 判定などと同様に、レシピ専用の解釈ロジックとして許容する、という判断です。汎用エンジン側の
語彙（`conditions`/`effects`）を、レシピのためだけに複雑化させないことを優先しました。

## 8. 設計原則のまとめ

- レシピは成果物の `object_defs` に、名前をキーとする辞書として定義する
- 製作中の状態は、成果物とは別の、自動生成された専用オブジェクトとして扱う。手書きの個別型にも、単一の汎用型にもしない
- 進捗は既存の `stages` を流用する。材料・道具の要求（`steps`/`requires`）は新設の専用スキーマとする
- 「箱」は既存の `slots` を `accepts` フィールドで拡張したものとして表現し、レシピ専用の新しいコンテナ概念にはしない
- 箱へのアイテム投入は、`accepts` から自動生成される `combinations` として実現する
- スロットの中身の検査・消費という具体的な処理は、YAML の汎用語彙を増やさず、プログラム側の専用ロジックとする

## 9. 未決事項・今後の検討課題

- 製作中オブジェクトの型の自動生成が、ゲーム起動時にエンジンがメモリ上で行う処理なのか、ビルド時に実際の YAML ファイルとして
  書き出す前処理ステップなのか
- 素材と道具を同じスロットにまとめるか、別スロットに分けるか
- 普通の箱（家具・コンテナ等）が `accepts` をどこまで再利用できるか（型・trait・サイズベースの緩い制約への拡張。
  普通のコンテナの合計サイズ制限・重さについては `ContainerSystem.md` で別途検討している）
- `steps.requires` の記法の詳細（`quantity` のデフォルト値、trait マッチングの可否など）
- `accepts` から自動生成される `combinations` で、複数の素材が同時にマッチしうる場合の優先順位（`ActionSystem.md` 6 節の
  既存の未決事項と関連）
- レシピの `icon` 以外に、自動生成される型へ引き継ぐべき情報（名前、説明文など）の範囲
- `slots` の容量制限・型制約という 7.1 節の既存 TODO は、本書の `accepts` 提案によって一部解決されるが、正式な仕様への
  反映は別途行う

## 10. 参考: 既存プロジェクト方針との整合性

- 製作中オブジェクトの「独立したプロセスオブジェクト」パターンは、`infection`（`GameElementDefinition.md` 11.3 節）・
  季節（`ClimateSystem.md`）と同型です。
- `accepts` によるスロットの拡張は、`GameElementDefinition.md` 7.1 節に残っていた既存の未決事項（slot の容量・型制約）の
  具体的な解決案です。
- 投入操作の自動生成は `ActionSystem.md` の `combinations` 設計をそのまま踏襲しています。
- 中身の検査・消費ロジックを汎用化しない判断は、`on_overflow` や地形の軸マッチングなど、既存の「専用スキーマ＋専用ロジック」
  という設計精神と一貫しています。
